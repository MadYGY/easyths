# 验证码识别引擎切换设计

## 背景

当前源码中的验证码识别统一使用 `ddddocr`。已安装的 `easyths` 模块中已经包含一套基于 SVM 的复杂验证码识别实现和 `captcha_svm.joblib` 模型资源，但这部分能力尚未合并回源码。

本次改动的目标是把复杂验证码识别能力最小化合并到源码中，同时保持简单数字验证码的默认行为不变。

## 目标

1. 配置文件中增加验证码识别类型开关。
2. 未填写或填写“数字验证码”时，继续使用 `ddddocr`。
3. 填写“复杂验证码”时，改用 SVM 模型识别数字加大小写字母验证码。
4. 复用现有 `BaseOperation -> get_captcha_ocr_server().recognize()` 调用链，不修改业务操作层接口。

## 非目标

1. 不合并安装包中的调试图片导出功能。
2. 不合并成功验证码样本保存和字符切分导出功能。
3. 不做自动识别验证码复杂度的策略判断。

## 方案比较

### 方案 A：配置驱动双引擎

在配置中显式指定验证码类型，OCR 层按配置选择 `ddddocr` 或 SVM。

优点：
- 行为确定，符合用户预期。
- 改动面小，对现有调用方透明。
- 便于写单元测试。

缺点：
- 用户需要显式配置复杂验证码场景。

### 方案 B：运行时自动探测复杂度

根据图片特征或 OCR 输出动态切换引擎。

优点：
- 用户配置更少。

缺点：
- 策略复杂且不稳定。
- 不符合本次需求中“把选项写在配置文件中”的约束。

### 方案 C：完整搬运安装包 OCR 模块

把安装包中的 `captcha_ocr.py` 整体拷贝进源码。

优点：
- 复用最多。

缺点：
- 会引入调试导出等本次不需要的附加行为。
- PR 范围膨胀，不利于审查。

## 选型

采用方案 A。

## 设计细节

### 配置

在 `ProjectConfig` 中新增 `captcha_type` 字段，并在 TOML 的 `[trading]` 段读取。

支持的业务值：
- 空值或未填写：按“数字验证码”处理，使用 `ddddocr`
- `数字验证码`：使用 `ddddocr`
- `复杂验证码`：使用 SVM

内部实现会把业务值归一化为稳定的引擎标识，避免 OCR 层直接依赖中文配置文案。

### OCR 路由

`CaptchaOCR.recognize()` 保持原有对外接口不变。

内部新增两条识别路径：
- `ddddocr` 路径：保留当前逻辑，作为默认实现
- `svm` 路径：从已安装模块提取最小必需实现，包括模型加载、图像抓取、前景裁剪、字符槽切分、HOG 特征提取和 SVM 预测

调用链仍然是：

`BaseOperation.ocr_captcha()` -> `get_captcha_ocr_server().recognize(control)`

### 资源与依赖

新增资源：
- `easyths/assets/captcha_svm.joblib`

新增依赖：
- `joblib`
- `opencv-python`

### 测试

新增单元测试覆盖：
- 配置文件未填或填“数字验证码”时，配置对象选择默认 `ddddocr`
- 配置文件填“复杂验证码”时，配置对象选择 `svm`
- `CaptchaOCR.recognize()` 按配置选择对应识别路径

测试通过 mock 图像抓取和模型调用实现，不依赖真实同花顺客户端。

## 风险与控制

1. `opencv-python` 为新增重量依赖
   控制方式：仅在 `server` 依赖组中增加，不影响纯客户端最小依赖

2. SVM 资源缺失导致复杂验证码模式不可用
   控制方式：把模型文件纳入包资源，并在复杂模式下明确抛出或记录加载失败

3. 现有业务调用误受影响
   控制方式：默认配置仍走 `ddddocr`，并保持公共接口不变
